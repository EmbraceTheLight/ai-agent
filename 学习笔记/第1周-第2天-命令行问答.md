## 当前输入逻辑

当前程序支持两个问题来源：

1. 命令行参数。
2. 标准输入。

优先级是：

```text
命令行参数 > 标准输入
```

这意味着如果运行时直接传入问题：

```powershell
go run ./cmd/assistant-cli "什么是 tool calling？"
```

程序会直接使用命令行参数作为用户问题。

如果没有传入参数，程序会提示用户输入问题，然后从标准输入读取一行内容。

这个设计让 CLI 同时适合两种使用方式：

- 快速一次性提问。
- 运行后再输入问题。

## 我对消息角色的理解

### system / developer

system prompt 可以理解为开发者给模型设置的长期行为规则。

它不代表用户这一次的问题，而是告诉模型：

- 你是谁。
- 你应该用什么风格回答。
- 你应该遵守什么边界。
- 你应该偏向什么领域或场景。

在当前代码中，这类规则通过 Responses API 的 `Instructions` 传给模型。

本项目中的定位是：

```text
你是一个帮助 Go 后端开发者学习 AI Agent 的技术助手。回答要准确、简洁、偏工程实践。
```

### user

user 表示用户本次真正提出的问题。

在当前 CLI 中，用户问题来自命令行参数或标准输入。例如：

```text
RAG 和普通搜索有什么区别？
```

这类内容是每次请求都会变化的部分。

### assistant

assistant 表示模型之前已经回答过的内容。

当前程序还不是多轮对话，所以暂时没有真正维护 assistant 历史消息。后续如果要做连续聊天，就需要把之前的用户问题和模型回答一起保存下来，再作为上下文传给模型。

## 遇到的问题

### 1. system prompt 应该放在哪里

一开始不清楚 system prompt 应该如何设置。

后来理解到，在 Responses API 中，可以使用 `Instructions` 表达这类全局行为设定；而在很多教程或 Chat Completions API 中，类似概念会被叫作 `system message`。

### 2. GoLand 和命令行环境变量不一致

运行过程中遇到过：

```text
401 Unauthorized
Invalid token
```

同一份代码在 GoLand 中可以运行，但在命令行中失败。

原因是 GoLand 的 Run Configuration 和终端中的环境变量不是同一套环境。最终通过确认并统一 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 后解决。

这个问题让我意识到：CLI 程序是否能运行，不只取决于代码，还取决于当前进程拿到的环境变量。

## 今日收获

1. CLI 程序应该把用户输入从代码中解耦出来。
2. 命令行参数适合一次性快速提问。
3. 标准输入适合运行后再输入内容。
4. system prompt 用来控制助手身份、边界和回答风格。
5. user message 才是用户当前真正的问题。
6. assistant message 主要用于后续多轮对话上下文。
7. IDE 和终端的环境变量可能不一致，排查 API 鉴权问题时要特别注意。

## CLI 还有哪些不方便

- 当前只读取了一个问题，还不是连续聊天。
- 标准输入目前只读取一行，不适合输入很长的多行问题。
- 输出格式还可以更统一，例如固定使用 `Assistant:` 前缀。
- `readQuestion` 后续可以更便于测试，例如内部统一使用传入的 `args` 参数，而不是直接依赖全局 `os.Args`。
- 错误提示可以继续打磨，让用户更清楚应该如何重新运行。
